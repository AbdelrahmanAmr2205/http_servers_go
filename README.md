# Chirpy - Go HTTP Server

A RESTful API backend for a Twitter-like application called "Chirpy", built as a guided project for the [Boot.dev](https://boot.dev) backend curriculum.

This project demonstrates how to build a robust, production-ready HTTP server in Go using the standard library (specifically leveraging Go 1.22+ routing features), PostgreSQL for data persistence, and custom middleware for authentication and metrics.

## Features

* **RESTful API**: Standardized endpoints for creating and managing users and "chirps" (messages).
* **Authentication & Authorization**:
* Secure password hashing using Argon2 (`argon2id`).
* JWT-based access tokens for stateless authentication.
* Stateful refresh tokens stored in the database.


* **Database Integration**:
* PostgreSQL database.
* Type-safe database interactions generated by `sqlc`.
* Schema migrations managed by `goose`.


* **Middleware**: Custom middleware for tracking metrics, incrementing file server hits, and authenticating users.
* **Profanity Filter**: Automatically redacts specific words from chirps before saving them.
* **Webhooks**: Endpoint for handling mock third-party webhooks (e.g., upgrading users to "Chirpy Red" via Polka).
* **Static File Serving**: Serves frontend HTML and assets from a local directory.

## 🛠️ Tech Stack

* **Language**: Go (Standard library `net/http`)
* **Database**: PostgreSQL
* **SQL Generator**: `sqlc`
* **Migrations**: `goose`
* **Authentication**: `golang-jwt/jwt`

## ⚙️ Getting Started

### Prerequisites

* **Go** (1.22 or higher)
* **PostgreSQL** (running locally or accessible via a URL)
* **`goose`** (for running database migrations)
* **`sqlc`** (optional, only if you plan to modify SQL queries)

### Installation

1. **Clone the repository**
```bash
git clone https://github.com/AbdelrahmanAmr2205/http_servers_go.git
cd http_servers_go

```


2. **Set up environment variables** Create a `.env` file in the root directory and configure the following variables:
```env
DB_URL=postgres://username:password@localhost:5432/chirpy?sslmode=disable
PLATFORM=dev
SECRET_KEY=your_super_secret_jwt_key
POLKA_KEY=your_mock_webhook_api_key

```


3. **Run Database Migrations** Navigate to the schema directory and run `goose`:
```bash
cd sql/schema
goose postgres "$DB_URL" up
cd ../..

```


4. **Build and Run**
```bash
go build -o chirpy
./chirpy

```


*The server will start on `http://localhost:8080`.*

## 📡 API Endpoints

### Health & Metrics

| Method | Endpoint | Description |
| --- | --- | --- |
| `GET` | `/api/healthz` | Check if the server is alive. |
| `GET` | `/admin/metrics` | View an HTML page with the total number of file server hits. |
| `POST` | `/admin/reset` | Reset metrics and drop all database tables (only available when `PLATFORM=dev`). |

### Users & Auth

| Method | Endpoint | Description | Auth Required |
| --- | --- | --- | --- |
| `POST` | `/api/users` | Create a new user. | No |
| `PUT` | `/api/users` | Update user credentials. | Yes (JWT) |
| `POST` | `/api/login` | Authenticate a user and receive access & refresh tokens. | No |
| `POST` | `/api/refresh` | Exchange a valid refresh token for a new access token. | No |
| `POST` | `/api/revoke` | Revoke a refresh token. | No |

### Chirps

| Method | Endpoint | Description | Auth Required |
| --- | --- | --- | --- |
| `POST` | `/api/chirps` | Post a new chirp (max 140 characters). | Yes (JWT) |
| `GET` | `/api/chirps` | Retrieve all chirps, ordered by creation date. | No |
| `GET` | `/api/chirps/{id}` | Retrieve a specific chirp by its UUID. | No |
| `DELETE` | `/api/chirps/{id}` | Delete a chirp (must be the author). | Yes (JWT) |

### Webhooks

| Method | Endpoint | Description | Auth Required |
| --- | --- | --- | --- |
| `POST` | `/api/polka/webhooks` | Upgrade a user to "Chirpy Red" status. | Yes (API Key) |

## 🗂️ Project Structure

```text
.
├── main.go               # Application entry point, server configuration, and route wiring
├── handler_*.go          # HTTP handler functions separated by domain
├── middlewares.go        # HTTP middleware functions (auth verification, metrics)
├── internal/
│   ├── auth/             # Cryptography logic (JWTs, password hashing)
│   └── database/         # Auto-generated sqlc code for database operations
├── sql/
│   ├── schema/           # goose migration files
│   └── queries/          # Raw SQL queries used by sqlc
└── index.html & assets/  # Static files served by the application

```
